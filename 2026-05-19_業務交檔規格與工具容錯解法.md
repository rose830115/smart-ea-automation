# Smart EA 自動化 — 業務交檔規格與工具容錯解法

> 起因：AYM_Syntex 案件（YIMS order_id=667）跑流水線時 `/api/orders/print/data` 回 500，追溯後發現是業務檔與公式範本內容偏離原始設計，腳本「忠實搬運」這些異常名稱送入 YIMS 導致後端炸掉。
> 目標：用「人的規範 + 工具最小幅度容錯」雙軌，避免再發生。

---

## 解法總覽（三層）

| 層 | 內容 | 對象 |
|---|---|---|
| 規範 A | 業務交檔規格 | 交檔的業務 |
| 規範 B | 公式範本填表規格 | 工具使用者（R&D 同事） |
| 工具改動 | 事前檢查面板、純數字 area name normalize、swab 編號 normalize | 既有 web 工具 |

---

## 規範 A：業務交檔規格

業務交給我們的 vendor Excel 檔，必須符合：

### A1. Sheet 名稱必須完全一致

只接受這三個 sheet name（**含大小寫**）：

- `Raw Material warehouse` ← warehouse 是小寫 w
- `Production Line`
- `Finished Goods Warehouse` ← Warehouse 是大寫 W

> 若業務改成 `Raw Material Warehouse`、`Production`、`FGW` 等變體，工具讀不到該 sheet，整個 zone 的資料會缺失。

### A2. Col A 的 checking point 必須照案件啟動時雙方議定的編號

- 合規範例：`3b`、`O1`、`O5`
- ❌ 不可改成功能名稱：`Cutting`、`Master Batch`、`Storage`
- ❌ 不可改成合併區編號：`1, 2 & 3`、`9 & 10`、`14 & 15`
- ❌ 不可自創代碼

> cp 是業務檔 ↔ 公式範本之間唯一的對應 key。cp 寫錯 → 工具完全找不到 area → 資料整行被跳過或誤判 zone。

### A3. swab 編號彈性由工具處理

不再強制業務照單一格式，工具會自動 normalize 常見變體（詳見「工具改動 3」）。但仍建議業務交檔時靠近標準（純數字或 `m1` 這類字母+數字格式）。

---

## 規範 B：公式範本填表規格

R&D 同事在建立公式範本（`target.xlsx`）的 `Areas` sheet 時：

### B1. Col D（area name）允許格式

- ✅ 純數字：`1`、`2`、`3` … （工具會自動補成 `Area 1`、`Area 2`）
- ✅ 標準連號：`Area 1`、`Area 2` …
- ❌ 文字命名：`Master Batch`、`Chips Storage`、`Finished Goods Storage`
- ❌ 含特殊字元：`1, 2 & 3`、`9 & 10`、`14 & 15`、`A+B`

### B2. Col F 必須填 RMW / PL / FGW

每個 area 都要明確指定 zone，**不可留空**。留空會讓 zone 走 fallback 邏輯，分區可能錯誤。

### B3. Col A / Col B / Col D / Col E 對應關係必須正確

- Col A：cp 編號（與業務檔 Col A 對應）
- Col B：該 cp 屬於哪個 report_area
- Col D：area name（assessment_area，送進 YIMS）
- Col E：該 area 的 function 描述

### B4. YIMS 後台 area 不需預先建立

工具會根據資料自動在 YIMS 後台新增 area，這部分維持現狀，不改變。

## 工具改動（既有 web 工具最小幅度修改）

### 改動 1：事前檢查面板

**位置**：web 介面上傳完業務檔 + 公式範本後、按「開始跑流水線」之前。

**檢查項目**：

| 項目 | 檢查內容 | 異常處理 |
|---|---|---|
| 業務檔 sheet 名稱 | 是否完全符合 A1 三個名稱 | 紅字列出缺失/多餘的 sheet |
| 業務檔 cp 格式 | 每行 Col A 是否符合 `^\d+[a-z]?$` 或 `^O\d+$` | 紅字列出「第 X 列 sheet `Y` cp 欄位異常：`Cutting`」 |
| 公式範本 Areas sheet Col D | 是否含禁用字元（中文字、`,`、`&`、`+`、空格內含等） | 紅字列出每個違規 area |
| 公式範本 Areas sheet Col F | 是否填了 RMW/PL/FGW | 紅字列出未填或填錯的 area |

**呈現**：
- 全部 pass → 顯示綠字「檢查通過，可以開始跑」+ 啟用「開始跑流水線」按鈕
- 任一 fail → 顯示紅字違規清單 + 禁用「開始跑流水線」按鈕，使用者必須選擇：
  - 退件給業務／工具使用者修正後重上傳
  - 直接在介面上手動修正（如果未來想做這個 feature，這份解法先不涵蓋）

**設計原則**：
- **只報錯不自動修正**：避免工具「猜錯」造成更難 debug 的問題
- **不動既有解析邏輯**：只在前面加一個檢查步驟

---

### 改動 2：公式範本 area name 純數字自動 normalize

**位置**：`smart_ea_automation.py` 的 `read_reference()` 函式內，讀 `Areas` sheet 時。

**規則**：
- Col D 值為純數字（int 或字串 `"1"`、`"2"` …）→ 自動轉成 `Area 1`、`Area 2`
- Col B（report_area）若同樣是純數字也一併 normalize（保持 cp ↔ report_area ↔ area name 三者一致）
- 其他值（標準格式 `Area 1`、或違規格式）→ 不動，違規的留給改動 1 報錯

**影響範圍**：
- 內部資料結構 `area_by_cp`、`area_info` 的 key 統一使用補完前綴後的名稱
- 後續 `build_backend_payload` 拿到的就是已正規化的 area name，送進 YIMS 不會踩特殊字元雷

---

### 改動 3：swab 編號 normalize

**位置**：`smart_ea_automation.py` 的 `cp_key()` 或新增 `normalize_swab_key()`，套用在業務檔讀進來時的 swab 欄位、以及公式範本菌落計數表的 swab 欄位比對處。

**標準格式定義**（依公式範本菌落計數表）：
- 主要：純數字 `1`、`2`、`3` …
- 次要：英文字+數字 `m1`、`m2` …

**業務檔變體 → 標準格式 mapping 規則**：

```
normalize 流程（虛擬碼）：
1. 全部轉小寫、去前後空白
2. 依序剝除前綴：「swab」「sw」「s」「#」「-」「_」「空白」
   - 注意要從長到短嘗試（先試 swab，再試 sw，再試 s），避免 sw1 被當成 s+w1
3. 若剩下純數字 → 回傳純數字
4. 若剩下「字母+數字」（如 m1）→ 保留（小寫）
5. 若剩下其他內容 → 回傳原始字串，並標記為「未識別」讓改動 1 在檢查面板報錯
```

**對照表**（驗收 normalize 規則時可用）：

| 業務寫法 | normalize 後 |
|---|---|
| `1` | `1` |
| `s1` | `1` |
| `S1` | `1` |
| `Swab-1` | `1` |
| `Swab1` | `1` |
| `SW 1` | `1` |
| `sw1` | `1` |
| `#1` | `1` |
| `m1` | `m1` |
| `M1` | `m1` |
| `cutting` | `cutting`（標記未識別） |

**設計原則**：
- normalize 只在「業務檔讀進來」與「跟公式範本菌落計數表比對」這兩處套用
- 公式範本端的 swab key 保持原樣（純數字 + m1 開頭），這樣標準命名不會被誤改
- 邏輯封裝在單一函式，未來新增變體只改一處

## 執行順序建議

1. **先寫兩份規範文件**（規範 A 給業務、規範 B 給 R&D 同事），放 `200_Reference/templates/`
   - 規範 A：交檔前發給業務，當作交檔協議
   - 規範 B：填公式範本時的填表指引
2. **接著實作改動 1（事前檢查面板）** — 風險最低、價值最高
3. **再實作改動 2（area name normalize）** — 一個函式
4. **最後實作改動 3（swab normalize）** — 一個函式 + 對照表驗證

## 回歸測試

AYM_Syntex（order_id=667）必須做為 regression case：
- 不修業務檔、不修公式範本，工具改動完成後重跑
- 預期：事前檢查面板紅字列出公式範本 Areas sheet Col D 的所有違規 area name（`Master Batch`、`Chips Storage`、`1, 2 & 3` 等）
- 使用者依紅字修公式範本（改成純數字或 `Area N`）後重跑 → 應該能順利打通 YIMS API 並拿到 risk_data
